# План унификации processor_script

## Цели

- Уйти от ручного написания десятков однотипных bash-обработчиков.
- Сделать сборку скриптов из переиспользуемых модулей (валидация входа, детект ошибок прав/файла, парсинг, сравнение, формат вывода).
- Обеспечить генерацию **без ИИ** как базовый режим.

## Базовый контракт (не ломаем)

- Вход: `CHECK_OUTPUT`, `ETALON_INPUT`.
- Выход: `Пройдена|Не пройдена|Ошибка`, `exit code: N`, опционально `ACTUAL_DATA:`.
- Карта кодов: `11–52` (источник истины — `backend/utils/error_codes.py`).

## Целевая архитектура

- **Script Core (library):** каталог модулей/блоков с единым интерфейсом `emit_bash(context) -> fragment`.
- **Rule Profiles:** декларативные профили проверок (JSON/YAML), описывающие «что проверяем».
- **Composer:** собирает итоговый bash из блоков, вставляет только нужные модули, гарантирует порядок (precheck -> parse -> assert -> output).
- **Validator:** статическая проверка профиля (обязательные поля, допустимые коды, совместимость блоков).
- **Generator CLI:** команда вида `generate-script --profile ... --target astra-linux` для не-ИИ генерации.

## Каталог модулей (минимальный набор)

- `precheck_env`: проверка `CHECK_OUTPUT`/`ETALON_INPUT` (51/52).
- `precheck_permission`: паттерны «нет прав» без ложных срабатываний (код 13).
- `input_empty`: обработка пустого вывода (41/или профильный код).
- `extractors`: 
  - `key_value_from_conf` (`key = value`),
  - `colon_table_field` (`getent passwd/group`),
  - `regex_capture`.
- `normalizers`: trim, split (comma/space/newline), dedup, case-sensitivity.
- `assertions`: contains_all, equals_set, equals_scalar, format_valid.
- `emitters`: единообразный финальный вывод (`Пройдена/Не пройдена/Ошибка`, `exit code`, `ACTUAL_DATA`).

## DSL профиля проверки (без кода bash)

- Обязательные секции:
  - `source` (тип вывода: `conf_file_line|getent_passwd|getent_group|raw_regex`).
  - `extract` (как достать фактические данные).
  - `expectation` (тип сравнения: contains/equals/regex).
  - `errors` (маппинг условий -> код 41/42/43/44/13/...).
  - `output` (`actual_data: true/false`).
- Примерно один профиль = один тип проверки; из профиля собирается скрипт.

## Совместимость с текущим UI

- Текущий `processor_script` остаётся исполняемым артефактом.
- В дальнейшем UI получает кнопку «Собрать из профиля», но на первой итерации только спецификация.
- Версионирование (`processor_script_version`) не меняется: генерируемый скрипт сохраняется как обычный processor_script.

## Контроль качества

- Эталонные fixture-наборы входов для каждого модуля: success/fail/permission/format edge-cases.
- Golden tests на generated bash (ожидаемый вывод + exit code).
- Запрет «опасных» паттернов в шаблонах (например, двусмысленный `sudo:` без контекста).

## План внедрения после архитектурного этапа

- Этап 1: спецификация блоков + DSL + примеры 10 типовых профилей (Astra Linux).
- Этап 2: реализовать composer + CLI генератор, покрыть тестами.
- Этап 3: миграция существующих вручную написанных скриптов в профили.
- Этап 4: опциональный UI-мастер в `ScriptsPage` (вызов генератора и предпросмотр).

## Ключевые артефакты спецификации

- ADR: почему выбрана декларативная генерация вместо ручного bash.
- Schema документа DSL.
- Каталог модулей с таблицей вход/выход/коды ошибок.
- Руководство «как добавить новый тип проверки без ИИ».
